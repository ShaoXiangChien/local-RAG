# Edge AI RAG Chat Demo: Take-Home Assignment

## 1. Project Overview

Build a Python-based chat application demo that supports Retrieval-Augmented Generation (RAG). The goal is to create an initial end-to-end prototype that demonstrates the full workflow from document ingestion, indexing, retrieval, and response generation.

This assignment is intended as a fast-moving prototype rather than a production-ready system. The priority is to prove that the RAG pipeline works locally and can be clearly demonstrated.

## 2. Core Requirements

### Application Goal

The application should allow a user to ask questions and receive answers generated by a local language model using retrieved context from local documents.

The demo should show:

- Loading supported local documents.
- Creating or updating a searchable vector store.
- Retrieving relevant document chunks for a user query.
- Generating an answer using the retrieved context.
- Running the full flow end to end in a demo-ready interface or command-line experience.

### Programming Language

- Python is required.

### Supported Data Sources

The document ingestion pipeline only needs to support:

- Markdown files (`.md`)
- Plain text files (`.txt`)

Additional file types are not required for the initial prototype.

## 3. Local-Only Execution Requirements

All model inference, embedding generation, document processing, retrieval, and response generation must run locally on-device.

Cloud-based services are not allowed for:

- Large language model inference.
- Embedding model inference.
- Vector search or document indexing.
- Any other core RAG pipeline step.

Examples of services that should not be used include:

- OpenAI API
- OpenRouter
- Any cloud-hosted LLM provider
- Any cloud-hosted embedding service

The implementation may use any local model serving approach, but the design choice should be explainable during the demo. For example, the developer should be able to explain why they selected a given local LLM runtime, embedding model, vector database, or retrieval strategy.

## 4. Technical Specifications

### Context Window Limit

The application must limit the LLM context window to approximately 6,000 tokens.

The implementation should account for:

- Retrieved document context.
- The user question.
- System or instruction prompts.
- Any response-generation constraints.

### RAG Implementation

There are no strict requirements for a specific RAG framework, algorithm, or open-source package. The implementation may use any reasonable local-first tools, libraries, or algorithms.

The initial version does not need advanced or highly specific optimization. The main requirement is that the RAG mechanism is functional and can be demonstrated clearly.

Possible implementation choices include, but are not limited to:

- Local embedding models.
- A local vector database or in-process vector index.
- Chunking and retrieval logic.
- A local LLM runtime.
- A simple chat interface or command-line interface.

## 5. Code Quality Expectations

The codebase should be clean, organized, and easy to understand.

The developer should be prepared to explain:

- The purpose of each major module or component.
- How documents are loaded and chunked.
- How embeddings are generated.
- How the vector store or retrieval layer works.
- How retrieved context is passed to the local LLM.
- How the 6k token context limit is enforced.
- Why specific tools, models, libraries, or architecture decisions were chosen.

The implementation does not need to be overly complex, but it should be structured enough to support discussion and future iteration.

## 6. Demo Expectations

The final prototype should be demo-ready and should show the complete RAG pipeline working end to end.

At minimum, the demo should demonstrate:

- Adding or loading local `.md` and `.txt` files.
- Building or loading the local index.
- Asking a question related to the indexed documents.
- Retrieving relevant context.
- Generating a local LLM response grounded in the retrieved content.

The demo should make it clear that the system does not rely on cloud services for model inference or embeddings.

## 7. Timeline

The requested timeline is ASAP, with a target of having an initial prototype ready by Friday of the same week the assignment was given.

The expectation is a working starting point, not a polished final product. Future iterations may improve quality, architecture, performance, or feature depth after the initial prototype is reviewed.

## 8. Follow-Up

After the prototype is ready, an initial demo meeting will be scheduled with Justin and potentially other engineers on the team.

The meeting will likely focus on:

- Walking through the working demo.
- Discussing technical design decisions.
- Explaining tradeoffs and implementation details.
- Identifying possible next steps or improvements.
- Meeting additional engineers from the team.

## 9. Success Criteria

The assignment is successful if the prototype:

- Runs fully locally.
- Uses Python.
- Supports `.md` and `.txt` document ingestion.
- Implements a working RAG pipeline.
- Limits the model context window to about 6k tokens.
- Uses no cloud-based LLM or embedding services.
- Is cleanly organized and explainable.
- Can be demonstrated end to end.
